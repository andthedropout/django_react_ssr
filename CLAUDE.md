# Project Guidance for Claude Code

This file gives coding assistants (Claude Code, Cursor, etc.) the context they need to work effectively in this repo. Read it before making changes.

## What this is

A **starter template** for production SaaS apps. Django REST API + React 19 + TanStack Start (real SSR) + Tailwind 4 + shadcn/ui, packaged as three lean Docker containers behind nginx.

**This template is the source of truth.** Forks should pull infra improvements from here via `MIGRATION.md` rather than diverging.

## Architecture (don't break these)

```
nginx (host port)
 ├── /api, /admin, /up, /media, /markdownx  →  web (Django gunicorn :8000)
 ├── /assets/, /static/, /favicon.svg, etc. →  served from baked-in nginx image
 └── everything else                         →  ssr (Bun + TanStack Start :3000) — REAL SSR

postgres + redis on docker network, NEVER exposed to host
```

Three single-purpose containers. **Don't merge them back into one** — that was the predecessor and the whole point of this rewrite was getting away from it.

## SSR is real — verify with curl, never trust comments

```bash
WEB_PORT=$(grep -E '^WEB_PORT=' .env | cut -d= -f2)
curl -s http://localhost:$WEB_PORT/ | grep -o 'SSR baked in'
# → "SSR baked in"  (text from <h1>, rendered server-side)
#
# If empty, the body has no rendered HTML and you're back in CSR mode.
```

The HTML response should contain rendered React content (`<h1>`, `<div>` with Tailwind classes, etc.) — NOT just `<div id="root"></div>`. Real SSR means Google bot, social link unfurlers, and users with slow JS all see the actual page.

## How the SSR pipeline works (the canonical TanStack Start setup)

```
vite build                                    # one command does it all
  ├── tanstackStart() plugin
  │     ├── generates frontend/src/routeTree.gen.ts from src/routes/*
  │     ├── builds the client bundle → .output/public/assets/*
  │     └── builds the SSR server bundle → .output/server/index.mjs
  └── nitro({ preset: 'bun' })
        └── inlines all server deps into .output/server/node_modules/
              (so the runtime container needs zero install — just bun + .output/)

Runtime:
  bun run .output/server/index.mjs           # SSR server starts, listens on :3000
  - request → renders React tree on the server → returns HTML + hydration data
  - client-side: <Scripts /> tag in __root.tsx loads the client bundle
  - hydrateRoot(document, <StartClient />) attaches React to the SSR'd DOM
```

## File responsibilities (don't reshape without reason)

| File | What it does | Don't do |
|---|---|---|
| `frontend/src/routes/__root.tsx` | Renders the **entire `<html>` document** (head + body + `<HeadContent />` + `<Scripts />`). Owns SSR shell. | Don't return just `<Outlet />` — that's the CSR pattern, breaks SSR. |
| `frontend/src/router.tsx` | Exports `getRouter()` factory used by client + server. | Don't add side effects at module scope (runs on both server + client). |
| `frontend/src/client.tsx` | `hydrateRoot(document, <StartClient />)` — hydrates the entire document. | Don't use `createRoot` or look for `#root`. There's no `#root` element. |
| `frontend/src/server.ts` | Delegates to `@tanstack/react-start/server-entry`. Thin wrapper. | Don't write your own static file server here (that's what we replaced). |
| `frontend/src/routeTree.gen.ts` | Generated by tanstackStart plugin from `src/routes/*` files. | Don't edit by hand — but DO commit it (Docker build needs it pre-existing for first build). |
| `frontend/vite.config.js` | tanstackStart + nitro({preset:'bun'}) + viteReact + tailwindcss plugins. | Don't add `rollupOptions.input` or `manualChunks` — TanStack Start owns those. Plugin order: tailwindcss → tanstackStart → viteReact → nitro. |
| `frontend/package.json` | `"build": "vite build"` only. No more `bun run build:server`. | Don't add a build:server step — TanStack Start + Nitro do it all. |
| `frontend/index.html` | **DOES NOT EXIST.** TanStack Start renders the `<html>` from `__root.tsx`. | Don't recreate it — having it would compete with the framework's HTML generation. |
| `frontend/server.ts` (root) | **DOES NOT EXIST.** | Don't write a custom Bun static file server. The original template had one and it never did real SSR despite its name. |
| `frontend/app.config.ts` | **DOES NOT EXIST.** Old TanStack Start used vinxi + app.config.ts. Current version (1.167+) is Vite Environments + nitro plugin only. | Don't recreate it. |

## Single source of truth: project name

`COMPOSE_PROJECT_NAME` in `.env` drives **everything** that's name-coupled:
- Docker volume names + network name
- Celery app name (`backend/config/celery.py` reads from env)
- Session/CSRF cookie names (`backend/config/settings.py` reads from env)
- Postgres user/DB name (matches `POSTGRES_USER` / `POSTGRES_DB`)
- Hashed base port that `bin/pick-ports` uses

To rename: `bin/rename-project new_name NewName`. Don't hardcode the project name anywhere new.

## Ports — never hardcode

All host ports come from `.env`:
- `WEB_PORT` — nginx (the only thing exposed to the host in prod)
- `DJANGO_PORT` — gunicorn directly (dev profile only)
- `VITE_PORT` — Vite dev server (dev profile only)

`bin/pick-ports` picks conflict-free values. If you find yourself writing `:8002` in code, use the env var instead.

## Common tasks

### Add a new Django app
```bash
docker compose --profile prod exec web python manage.py startapp my_app
```
Then:
1. Move it under `backend/` if not already there
2. Add `'my_app'` to `INSTALLED_APPS` in `backend/config/settings.py`
3. Add its URLs to `api_patterns` in `backend/config/urls.py`
4. Run `python manage.py makemigrations my_app && python manage.py migrate`

### Add a new shadcn component
```bash
docker compose --profile dev exec js bunx shadcn@latest add tooltip
```
**The package is `shadcn` (not `shadcn-ui` — that's deprecated).** New components land in `frontend/src/components/ui/`.

### Add a new SSR-rendered route
```tsx
// frontend/src/routes/my-page.tsx
import { createFileRoute } from '@tanstack/react-router';
import MyPage from '@/pages/MyPage';

export const Route = createFileRoute('/my-page')({
  loader: async () => ({ /* server-side data */ }),
  component: MyPage,
});
```
SSR is on by default. Verify: `curl http://localhost:$WEB_PORT/my-page` should return rendered HTML for the page content, not an empty shell.

### Add a new theme
1. Drop `my-theme.json` into `design-system/themes/`. Match the structure of an existing one — copy `vercel.json` as a starting point.
2. Set `VITE_FRONTEND_THEME=my-theme` in `.env`.
3. Done. The file is auto-served at `/static/themes/my-theme.json`.

## Build & rebuild

**Always use `bin/rebuild`** — it enforces the ssr + nginx pairing:

```bash
bin/rebuild              # rebuild + restart ssr + nginx (frontend changes)
bin/rebuild web          # rebuild + restart web only (backend changes)
bin/rebuild all          # rebuild + restart everything
```

⚠️ **Critical:** `ssr` and `nginx` MUST be rebuilt together. Both consume the same `frontend-build` stage; the content-hashed asset filenames (`index-Bzr54oq8.css`) change every build. If you rebuild only one, the SSR'd HTML references hashes nginx doesn't have → all `/assets/*` 404 → blank page.

If you must use raw `docker compose`:
```bash
docker compose --profile prod build ssr nginx          # frontend rebuild — both, always
docker compose --profile prod build web                # backend-only — web alone is fine
```

If a build is slower than 30 sec for a code-only change, BuildKit cache mounts probably broke. Check that the Dockerfile still has `--mount=type=cache,...` lines.

## Lockfile management

- **Backend:** `requirements.txt` is the source. uv installs from it.
- **Frontend:** `frontend/bun.lock` is regenerated when `package.json` changes. If you bump deps, delete the lockfile before rebuilding so Bun regenerates it.

## SSR-specific gotchas (things to test for in custom code)

1. **No `window`/`document`/`localStorage` at module scope** — any code that runs at import time runs on the server too. Wrap browser-only code in `useEffect` or guard with `typeof window !== 'undefined'`.

2. **Hydration mismatches** — server-rendered HTML must match what the client first renders. Common causes:
   - Random IDs / `crypto.randomUUID()` in render
   - Time-based content (`new Date().toLocaleString()`)
   - Reading `localStorage` during render
   - Conditional UI based on `window.matchMedia()`
   Symptoms: console error "Hydration failed" + bits of UI flicker. Fix by moving the conditional to `useEffect` or using `<ClientOnly>` from `@tanstack/react-router`.

3. **Theme FOUC** — `useTheme()` fetches the theme JSON in a client-side `useEffect`. SSR renders with the fallback theme; on hydration it swaps. If the swap is jarring, pre-fetch the theme in a root loader and pass via context.

## Things NOT to do (lessons we've already learned the hard way)

1. **Don't write a custom `frontend/server.ts`** — TanStack Start + Nitro generate `.output/server/index.mjs`. Custom server.ts that loads dist/ as static files is what made the OLD template look like SSR while actually being CSR.
2. **Don't recreate `frontend/index.html`** — TanStack Start renders the `<html>` document from `__root.tsx`. Having both competes and breaks the build.
3. **Don't recreate `frontend/app.config.ts`** — old vinxi-era API. Current setup is all in vite.config.js via plugins.
4. **Don't make `__root.tsx` return `<Outlet />` alone** — must render `<html><head><HeadContent/></head><body><Outlet/><Scripts/></body></html>`.
5. **Don't use `createRoot` in `client.tsx`** — use `hydrateRoot(document, <StartClient />)`. There's no `#root` element to mount on.
6. **Don't add `rollupOptions.input` to vite.config.js** — TanStack Start owns the entry config.
7. **Don't add `manualChunks` as an object** — Vite 8/Rolldown wants a function. We're on Vite 7 for now, but be aware if/when bumping.
8. **Don't disable the TanStack Start plugin in production builds** — that's what made the predecessor "fake SSR".
9. **Don't bump to Vite 8 yet** — plugin ecosystem isn't ready (transformWithEsbuild deprecation breaks `@tailwindcss/vite`). See MIGRATION.md Tier 4.
10. **Don't use `\b` in BSD `sed`** on macOS — use `perl -pe` instead.
11. **Don't expose postgres/redis to the host** — they're internal-only by design. If you need direct DB access, use `docker compose exec postgres psql ...`.
12. **Don't commit `.env`** — it's gitignored. Use `.env.example` for templated values.
13. **Don't reintroduce supervisord** — the predecessor had nginx + supervisord + gunicorn + bun-ssr in one container. We split that. Keep it split.

## Performance baseline (verify your changes don't regress)

```
Image sizes:    nginx 33 MB,  ssr 283 MB,  web 362 MB        (~678 MB total)
RAM at idle:    nginx 6 MiB,  ssr 59 MiB,  web 78 MiB,       (~165 MiB total)
                postgres 19 MiB, redis 4 MiB
Cold build:     ~2 min
Warm rebuild:   ~10 sec
Build context:  ~10 MB (.dockerignore is doing its job)
SSR verified:   curl returns rendered <h1> + <div> + content (not empty shell)
```

If your change makes any of these significantly worse — or if SSR stops returning rendered content — dig in before merging.

## Testing the stack

```bash
WEB_PORT=$(grep -E '^WEB_PORT=' .env | cut -d= -f2)

# Sanity
curl -sf http://localhost:$WEB_PORT/up && echo "django ok"
curl -sI http://localhost:$WEB_PORT/ | grep "200"

# Real-SSR check — response body should contain actual rendered content
curl -s http://localhost:$WEB_PORT/ | grep -o 'SSR baked in'
# Should print "SSR baked in" — that text comes from <h1> on the homepage.
# Empty result = SSR broken, you're back in CSR.

# Disk hygiene
docker system df
du -sh /var/lib/docker/containers/*/

# Live RAM
docker stats --no-stream --format "table {{.Name}}\t{{.MemUsage}}" \
  $(docker ps --filter "name=$(grep '^COMPOSE_PROJECT_NAME=' .env | cut -d= -f2)" --format "{{.Names}}")
```
